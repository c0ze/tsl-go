# Web front-end via WebAssembly (23a) Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Play tsl-go in a browser: a `js/wasm` front-end behind the same
`ui.Prompter`/`ui.Renderer` seam the terminal uses — zero engine changes —
published to GitHub Pages at **tsl.arda.tr**.

## Design
- `ui.ActionForRune(r)` — the key map extracted from the tcell switch so both
  front-ends share one binding table (tcell keeps its arrow/Enter handling).
- `internal/ui/web/render.go` (untagged, pure, **testable**): `RenderHTML`
  (the View as `<pre>` spans — color classes per the tcell palette, dim runs,
  HTML-escaped glyphs, optional target cursor) and `MenuHTML`.
- `internal/ui/web/web.go` (`js && wasm`): keydown events feed a channel;
  `NextAction`/`Menu`/`Target` block on it (Go's wasm scheduler yields to the
  JS loop); Render writes innerHTML. Menu/Target mirror the terminal UX.
- `internal/boot`: the game constructor extracted from `cmd/tsl` so both
  mains share it. `cmd/tsl-wasm` (`js && wasm`): saves go to
  `localStorage` — resume-and-delete on boot, "Game saved. Reload to
  resume." on S — the same no-scumming semantics as the file.
- `web/`: `index.html` (black, monospace, the palette as CSS), `CNAME`
  (tsl.arda.tr).
- CI: ci.yml gains a `GOOS=js GOARCH=wasm` build check; new `pages.yml`
  builds the site (wasm + `wasm_exec.js` from GOROOT) and deploys to Pages
  on every master push. Pages switched to workflow builds via the API; DNS
  (CNAME tsl.arda.tr → c0ze.github.io) is the user's side.
